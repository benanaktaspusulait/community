# Phase 1 — Pilot Migration

## Goal of the Phase

Phase 1 ships a first version strong enough to migrate the core needs of **1–3 existing WhatsApp, Facebook, or Telegram groups** into this product.

Phase 1 is not about digitizing all community life. It focuses on:

- making old knowledge **findable**
- making topic-based groups **manageable and disciplined**

## Target Users

- WhatsApp/Facebook Group/Telegram admins
- very active community members (“power users”)
- newcomers who frequently ask questions
- members posting jobs/housing/services/recommendations
- users who want fast outcomes via structured **help requests**

## First Pilot Community and Hero Use-Case

Suggested first pilot:

- **Community type**: UK Turks
- **Scope**: one dense location or a city cluster
- **Hero request**: “looking for a room / housing”
- **Supporting request**: “looking for a provider / recommendation”

Why this works:

- housing/room requests repeat frequently
- creates a fast “aha moment” for newcomers
- community memory + search becomes useful immediately
- provider demand creates an early commercial surface (later phases)

## Product Thesis (Phase 1)

> Turn the most repeated needs in a community into **structured + searchable** artifacts instead of chat noise.

## Phase 1 Success Definition

Phase 1 is successful if:

- admins can share invite links and start migration
- users can **search** and find old answers
- the `community > location > topic group` structure is clear and enforceable
- repeat-question rate starts decreasing
- at least a few groups begin shifting daily activity here
- reporting and basic moderation flows work end-to-end
- a new user’s first session results in either:
  - finding an answer, or
  - successfully creating a structured help request

## Aha Moment (First 1–2 Minutes)

Users should feel:

- “The answer was already asked here.”
- “If not, I can open it in a clear structured format.”
- “Admins run this place with less chaos.”

Therefore onboarding/first screens must not feel empty and should highlight:

- solved threads
- popular help requests
- pinned knowledge/resources
- relevant location/topic groups

## Content Strategy (Curated-First)

During the pilot, content must not be fully organic:

- **curated-first**: admins/mods/migration concierge build the initial useful library
- **organic-second**: member contributions come on top
- **memory-first**: success is fast outcomes, not high post volume

## Scope (What Phase 1 Must Include)

### Core Pilot (must work)

- community / location / topic group structure
- invite + join request + (optional) preview entry points
- structured creation flows (help request, question, listing) backed by templates
- thread + comment/reply model (async)
- search + filters + “duplicate deflection” before posting
- library/knowledge cards (seeded + curated)
- basic moderation: report, remove content, remove member (scoped)
- user ad visibility preference (show/hide)

### Launch Guardrails (must exist, can be minimal)

- account deletion path and basic data request process (can be manual/semi-manual)
- rate limiting + spam signals
- ad approval basics (if ads are enabled in the pilot)
- deep links + preview behavior (shared threads/resources)
- bilingual UI baseline (if the pilot requires it)

## Explicit Non-Goals (Out of Phase 1)

- full chat and DMs
- payments/escrow/booking/delivery
- full rating/review system
- full event module
- polling/voting engine
- heavy identity verification
- ad billing / open auction model
- deep analytics and growth automation
- enterprise sales configuration

## Expected Outcome at the End of Phase 1

At the end of this phase, the product should be describable as:

> A community platform entered via invite, preview, or join request; organized by topics; building memory via async threads; searchable; and with ad visibility controlled by the user.

If this is clearly working, moving to Phase 2 is justified.
